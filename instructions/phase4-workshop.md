# Phase 4: ワークショップ機能

## 目的

Phase 2/3 で「個人がプレイして実感を得る」装置ができている上に、**ワークショップ運営に必要な機能**を載せる。具体的には複数シナリオ、複数台同時シミュレーション、統計出力、共有可能URL。

## 着手の前提

- Phase 2 / Phase 3 の受入基準を全て満たしていること
- 体験レイヤが動作しており、個人プレイで実感が得られる状態
- core/ の純粋関数性が保たれていること(これがないと統計・バッチ実行ができない)

## このPhaseで実現すること

ワークショップでの3つの典型的な使い方をサポートする:

### 使い方1: 個人プレイ → 全員で振り返り

各参加者がそれぞれ1台プレイ → 終了後に全員のログを集計表示 → 「人によってこんなに違う一生になる」を体感

### 使い方2: シナリオ比較

「日本標準 vs ASEAN輸出 vs ライドシェア」の3シナリオを並べてプレイ → ステークホルダ関与の違いを比較

### 使い方3: バッチ統計

100台ランダムプレイを事前に実行 → ヒートマップ・分布・キャプティブ介入機会の統計を提示

## スコープ

### 1. シナリオ選択UI(ScenarioSelector.tsx)

- スタート画面で `data/scenarios.json` のシナリオを選択
- 各シナリオのカード表示(名称・description・narrative_hint・主要パラメータ)
- 選択後にプレイ画面に遷移

### 2. シナリオ追加・編集機能(オプション)

- 「カスタムシナリオを作る」ボタン
- 既存シナリオをコピーして確率倍率や初期パラメータを編集
- 作成したシナリオはローカルストレージに保存
- インポート/エクスポート(JSON)

### 3. 統計ビュー(StatsView.tsx)

プレイ完了時または「統計を見る」ボタンで表示。以下のセクション:

#### 3.1 基本統計
- 総イベント発火数(カテゴリ別)
- 経過日数 / 走行距離 / 最終価値
- 関与STK数 / レイヤ別関与数

#### 3.2 ステークホルダ関与ランキング
- 登場回数Top10
- 役割別ヒートマップ(STK x 役割の頻度行列)

#### 3.3 状態変化軸の累積
- 7軸それぞれの発火回数
- 軸ごとのタイムライン

#### 3.4 キャプティブ介入機会(副目的γ)
- STK02が起点になっているイベント数
- STK02が関与可能だが関与しなかったイベント数(機会損失)
- L6/L1のイベントでSTK02不在のもの一覧

#### 3.5 比較ビュー(複数シナリオ実行時)
- 同じ車両を複数シナリオでプレイした結果の並列表示
- レーダーチャートで7属性の最終値比較

### 4. バッチランナー(src/cli/batch.ts)

```bash
npm run sim:batch -- --scenario=default-japan-2026 --count=100 --seed-base=1000 --out=results.json
```

- 100台分のランダムプレイを並列実行
- 各台のログを results.json に保存
- 集計サマリーをstdoutに表示

このバッチ結果をUI側で読み込んで統計ビューに表示する機能も追加(`StatsView.tsx` に「バッチ結果を読み込む」ボタン)。

### 5. 共有可能URL(QueryString方式)

プレイの状態(シナリオ・シード・選択履歴)を URL クエリパラメータにエンコード:

```
https://example.com/?s=default-japan-2026&seed=42&decisions=EV007,EV010,EV012,...
```

- 「このプレイを共有」ボタンでURLをクリップボードコピー
- 共有URLを開くと同じ状態が再現される(同じシード+同じ決定 = 同じ結果)
- 決定の数が多い場合は LZ-string で圧縮

### 6. ログのエクスポート/インポート

- プレイ中・プレイ後にLedgerをJSONエクスポート
- 過去のJSONログをアップロードして可視化のみ行う(リプレイモード)

### 7. Workshop Mode(オプション)

複数の参加者が個別にプレイした結果を1画面に集約表示するモード:

- ファシリテータがシナリオを指定
- 参加者は個別URLでアクセス
- ファシリテータ画面に各参加者のログが届き、集計される
- リアルタイム同期は不要(参加者がプレイ完了時にエクスポートしたJSONをファシリテータが受け取る形でよい)

## 非ゴール(Phase 4 でも実装しない)

- サーバサイド永続化(Firebaseなど) → 全てローカルストレージ + JSONエクスポートで完結
- ユーザ認証・ログイン → 不要
- リアルタイム同期(WebSocket等) → 不要
- 商用展開機能(課金、広告) → 全くの対象外
- スマートフォン最適化 → デスクトップ前提でよい(タブレット表示は可)

## 受入基準

1. シナリオ選択画面でデータ駆動的に5つのシナリオが選べる
2. 選択後に当該シナリオのパラメータでプレイが始まる
3. プレイ完了後に統計ビューが表示され、上記4セクションが全て見える
4. キャプティブ介入機会セクションで、機会損失イベントが具体的に列挙される
5. `npm run sim:batch -- --count=100 --scenario=default-japan-2026` がエラーなく完了し、results.json が生成される
6. 生成された results.json を UI で読み込んで統計ビューが表示される
7. 「共有URL」ボタンでURLが生成され、別ブラウザで開くと同じ結果が再現される
8. ログJSONエクスポート → 別タブでインポートしてリプレイ表示できる
9. Phase 2/3 の機能が全て動作している

## 推奨実装順序

### Step 1: シナリオ選択UI

`scenarios.json` から読み込んでカード表示。useSimulation の引数として scenarioId を渡せるようにする。

### Step 2: バッチランナー

CLI実装。core/ が純粋関数なのでそのまま並列実行可能(Promise.all + workerThreads 不要、シングルスレッドでも100台なら数秒)。

### Step 3: 統計ビュー(基本)

3.1〜3.3 のセクション。Recharts などでグラフ化。Phase 3 で導入したライフラインビューを再利用できるところは再利用。

### Step 4: キャプティブ介入機会の集計ロジック

`core/selectors.ts` に新規関数追加(`computeCaptiveOpportunities`等)。これはロジック層なので core/ に書く。

### Step 5: 共有URL

LZ-string で圧縮 → btoa → クエリパラメータ。デコード時の逆変換。

### Step 6: ログ入出力

エクスポートはダウンロードリンクでJSONを保存。インポートはファイル選択でローカル読込み。

### Step 7: 統計ビュー(比較)

複数シナリオ実行時のレーダーチャート等。やや手間がかかるので最後でよい。

### Step 8: Workshop Mode(余裕があれば)

ファシリテータ画面と集計表示。実装難度は高くないが、UIデザインに時間がかかる。

## 設計上の注意

### core/ への追加は selectors.ts に集中

統計ビューの集計ロジックは core/selectors.ts に書く(派生情報の計算)。UI側に集計を書かない。

```typescript
// core/selectors.ts
export function computeStakeholderRanking(ledger: Ledger): Array<{ stakeholder_id: string; count: number; roles: Record<string, number> }>;
export function computeRoleHeatmap(ledger: Ledger): number[][];
export function computeCaptiveOpportunities(ledger: Ledger, allEvents: EventDef[]): {
  participated: number;
  missed: Array<{ event_id: string; reason: string }>;
};
```

### バッチ実行のパフォーマンス

100台プレイは数秒で終わる想定。10000台でも分単位。それ以上ならWeb Worker化を検討。

### URL圧縮の限界

決定履歴が長くなると URL が長くなる。1000イベント超になると一般ブラウザの URL 長制限(~2KB)に達する可能性。その時は localStorage キーに保存して URL では key だけ共有する代替策。

### 既存機能との整合

Phase 4 の機能追加で Phase 2/3 の機能が壊れないことを優先する。既存テストを通したまま追加するのが原則。
