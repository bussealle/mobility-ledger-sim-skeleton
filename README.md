# Mobility Lifecycle Simulator (MLS)

車両15年ライフサイクルを「車両=ステートマシン」「イベント=トランザクション」「ステークホルダ=役割を持つ関与主体」として表現し、プレイヤーが選択を重ねることで1台の車両の人生を構成するインタラクティブ・シミュレータ。

## このプロジェクトの目的

ワークショップで車両ライフサイクルとステークホルダ関係を**実感を伴って体験する**ための装置。具体的には:

- **主目的(α 教育型)**: 1台の車に、いかに多くの主体・トランザクション・状態変化が累積するかを体験的に理解する
- **副目的(γ 戦略型)**: キャプティブファイナンスの介入機会を、累積ログと統計から動的に発見する

派生効果として、データ整合性チェック(到達不能イベント・矛盾の検出)も得られる。

## Claude Codeでの進め方

1. **必ず `docs/` を上から順に読む**(設計思想 → 状態モデル → イベントモデル → STKモデル → アーキテクチャ → 用語集)。これを飛ばすと中身を理解せずに「もっともらしい」コードを書いてしまうので絶対に飛ばさない。
2. `data/*.json` を一度すべて目を通し、データ構造を把握する。
3. 着手するPhaseの `instructions/phaseN-*.md` を読み、受入基準と非ゴールを確認する。
4. 該当Phaseのスコープ内で実装する。スコープ外には手を出さない。
5. 受入基準を満たしたら、その時点で停止して報告する。

## 技術スタック

- **言語**: TypeScript(strict)
- **UI**: React 18 + Vite
- **スタイル**: Tailwind CSS(または vanilla CSS Modules、好みに応じて)
- **テスト**: Vitest
- **状態管理**: ロジック層は純粋関数、UI層はカスタムフックで管理(Reduxやxstateは使わない)

ロジック層は**純粋なTypeScript関数の集合**で、Reactに依存しない。UI層は**ロジック層を呼び出して結果を表示するだけ**。この分離は厳格に守る(理由は `docs/05-architecture.md` 参照)。

## Phase概要

| Phase | スコープ | 受入基準の核 |
|-------|---------|-------------|
| Phase 2 | コアエンジン + 最小UI | プレイヤーが選択を進めて廃車まで到達でき、ログが残る |
| Phase 3 | 体験レイヤ | ステークホルダ星座・車両オーラ・トランザクション展開で「実感」が出る |
| Phase 4 | ワークショップ機能 | シナリオ選択・複数台同時実行・統計出力・共有可能URL |

各Phaseは**前Phaseが完成していることを前提**に積み上げる。Phase 2が動く前にPhase 3に進まない。

## 設計上の重要な決定

- **車両は単一変数では表現できない**: 「相 × 7属性ベクトル」のステートマシンとして扱う(詳細は state-model.md)
- **イベントは制約付き状態遷移関数**: 事前条件・事後効果・トリガー型(choice/probabilistic/time/cascade)を持つ
- **ステークホルダは役割で関与**: 起点 / 対象 / 実行 / 確認 / 受益 / 責任 の6機能軸
- **6レイヤ構造**: 50主体は行動プロファイルからクラスタリングで6レイヤに整理されている(L1〜L6)
- **時間モデルはハイブリッド**: 通常はイベント駆動、年次節目で強制ステップ
- **完全性より体感**: 第1版では精度6割でも体験として成立すれば良い。事前条件の組み合わせ爆発は許容しない
