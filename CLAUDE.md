# Claude Code 向けナビゲーション

このファイルはあなた(Claude Code)がこのプロジェクトに入ったときに最初に読むファイルです。
プロジェクトの全体像と進め方を最短で把握するためのナビゲーションを提供します。

## まず最初に

ここはまだ実装が始まっていないプロジェクトのスケルトンです。
あなたの仕事は **指示された Phase の実装を行うこと** ですが、その前に必ず以下を読んでください:

1. `README.md` (プロジェクト全体像)
2. `docs/` 配下を **全て**(6ファイル, 順番通り)
3. `data/` 配下の JSON を **全て** 一度ざっと確認
4. 着手する Phase の `instructions/phaseN-*.md`

**読まずに着手することは禁止します**。docs と data には、コード上の表面からは読み取れない設計判断が大量に含まれています。

## このプロジェクトの一行要約

車両15年ライフサイクルを「車両=ステートマシン」「イベント=トランザクション」「ステークホルダ=役割を持つ関与主体」として表現し、プレイヤーが選択を重ねることで1台の車両の人生を構成するインタラクティブ・シミュレータ。**ワークショップ用の体験装置**であり、実用システムではない。

## 絶対に守るべき原則(失敗事例の予防)

### 1. ロジック層とUI層の厳格な分離

`core/` (純粋TypeScript) と `ui/` (React) は独立。`core/` は Node.js 単独で動く必要がある。React Hooks の中に状態遷移ロジックを書かない。Component 内で確率判定を書かない。詳細は `docs/05-architecture.md` を読むこと。

### 2. 設計思想を曲げない

主目的は **(α) 教育型 — 体験的理解**。副目的が **(γ) 戦略型 — キャプティブ介入機会発見**。それ以外(完全シミュレーション、ゲーム性、商用展開)は非ゴール。「便利そうだから」という理由で機能追加しない。

### 3. Phase の境界を守る

Phase 2 の指示書には Phase 3/4 の機能を入れない。たとえ「ついでに作れる」と思っても、Phase 境界を越えない。各 Phase の指示書には **非ゴール** が明記してある。

### 4. 完全性より体感

第1版では精度6割で良い。事前条件の組み合わせ爆発を解こうとしない。「動くこと」を優先する。`docs/03-event-model.md` の「実装上の注意」を参照。

### 5. 履歴を消さない

`prior_owners`, `accident_history`, `service_history`, `attestations` 等の履歴系フィールドは追記のみ。削除しない。これは設計上の不変条件。

## 推奨される作業の進め方

### Phase に着手する初日

```
1. README.md を読む (5分)
2. docs/ を順に読む (30〜45分)
3. data/*.json をざっと眺める (10分)
4. instructions/phaseN-*.md を読む (10分)
5. 受入基準を再読し、自分の言葉で要約してみる (5分)
6. 実装計画を立てる (15分)
7. 実装着手
```

最初のドキュメント読みに 1 時間以上使うことを許容してください。それを飛ばすと後で確実に時間を失います。

### 不明点があるとき

- 用語が分からない: `docs/06-glossary.md` を引く
- データの意味が分からない: 元の xlsx (`vehicle_lifecycle_workshop_v4.xlsx`) を user に求める
- 仕様が決まっていない部分: 自分で決めて良いが、判断基準は **「主目的(α)に貢献するか」**
- 技術選定で迷う: 各 Phase 指示書の「設計上の注意」と「推奨実装順序」を参照

### 進捗の報告

各 Phase の完了時には、その Phase 指示書末尾の **完了報告フォーマット** に従って報告してください。

## やってはいけないこと

- docs/ と data/ を読まずに着手
- Phase 境界を越えた機能追加
- core/ から ui/ や DOM API への依存追加
- `Math.random()` の使用 (シード可能PRNGを使う)
- 履歴フィールドの削除
- 美術的に凝った UI への注力(機能優先)
- 商用展開・課金・認証・サーバ永続化の検討

## 推奨する技術スタック

`docs/05-architecture.md` 参照。要約:

- TypeScript (strict)
- React 18 + Vite
- Tailwind CSS or vanilla CSS Modules(自由)
- Vitest
- 状態管理は Redux/xstate を **使わない**(useState + custom hook で十分)

## 元データの根拠

このプロジェクトのデータは別途存在する Excel ワークブック `vehicle_lifecycle_workshop_v4.xlsx` から自動生成されています。
データの意図や根拠を確認したいときは、user にこのファイルを参照してもらってください。

128イベント・50主体・463リンク。各イベントには状態変化軸(7+1)が、各主体には行動プロファイルから帰納的に抽出された6レイヤ(L1〜L6)が付与されています。
